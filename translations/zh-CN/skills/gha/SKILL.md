调查这个 GitHub Actions URL：$ARGUMENTS

使用 gh CLI 分析此工作流运行。你的调查应当：

1. **获取基本信息并识别实际失败**：
   - 哪个工作流/作业失败了，发生在何时，以及对应哪个提交？
   - 关键：仔细阅读完整日志，找出具体是什么导致了 exit code 1
   - 区分警告/非致命错误与实际失败
   - 查找如 “failing:”、“fatal:” 或决定何时 exit 1 的脚本逻辑等模式
   - 如果你同时看到 “non-fatal” 和 “fatal” 错误，聚焦于真正导致失败的原因

2. **检查是否存在偶发性（flakiness）**：检查**完全相同的失败作业**最近 10-20 次运行：
   - 重要：如果一个工作流有多个作业，你必须检查失败的**特定作业**的历史，而不只是工作流
   - 使用 `gh run list --workflow=<workflow-name>` 获取 run ID，然后用 `gh run view <run-id> --json jobs` 检查特定作业状态
   - 这是一次性失败，还是这个**特定作业**的重复模式？
   - 该**作业**最近的成功率是多少？
   - 这个**作业**上次通过是什么时候？

3. **识别引入问题的提交**（如果该特定作业存在失败模式）：
   - 找到该**特定作业**首次失败的运行，以及它最后一次通过的运行
   - 识别引入失败的提交
   - 通过检查进行验证：该**作业**在该提交之后的所有运行中都失败吗？在该提交之前的所有运行中都通过吗？
   - 如果验证成立，请以高置信度报告该问题提交（breaking commit）

4. **根本原因**：基于日志、历史以及任何问题提交，最可能的原因是什么？
   - 聚焦于**真正**导致失败的原因（而不只是你看到的任何错误）
   - 根据日志和失败逻辑验证你的假设

5. **检查是否已有修复 PR**：搜索可能已解决此问题的开放 PR：
   - 使用 `gh pr list --state open --search "<keywords>"`，关键词可使用相关错误信息或文件名
   - 检查是否有任何开放 PR 修改了失败文件/工作流
   - 如果存在修复 PR，在报告中注明并跳过建议部分

撰写最终报告，包含：
- 失败摘要（具体是什么触发了 exit code 1）
- 偶发性评估（一次性 vs 重复发生，成功率）
- 问题提交（若已识别并验证）
- 根本原因分析（基于**实际**失败触发因素）
- 现有修复 PR（若找到——包含 PR 编号和链接）
- 建议（如果已存在修复 PR，则跳过）
